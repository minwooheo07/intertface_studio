# API 테스터 (포스트맨식)

## 목적
업체에서 API 스펙을 받았을 때, 인터페이스 정의를 만들기 **전에** "이 API가 실제로 이렇게 응답하는가"를
직접 두들겨보는 도구. 사이드바 [API 테스트] 메뉴.

포스트맨 대신 여기서 하는 이점:
- **CORS 없음**: 브라우저가 아니라 연계서버(백엔드)가 대신 호출한다.
- **사내망 API 테스트 가능**: 연계서버가 접근 가능한 내부 API도 그대로 호출된다.
- 인터페이스 정의와 같은 화면 안에 있어 스펙 확인 → 정의 등록 흐름이 이어진다.

## 사용법
1. 메서드(GET/POST/PUT/DELETE/PATCH) 선택, URL 입력
2. 필요하면 헤더 추가 (Authorization 등)
3. 바디 타입 선택: NONE / JSON / FORM(x-www-form-urlencoded) / RAW
4. [전송] → 응답 상태·소요시간·바디·헤더 확인

## ⚠️ 실제 호출임 (시험실행과 다름)
- 인터페이스 편집기의 **시험실행**은 절대 전송하지 않는 미리보기다.
- 반면 **API 테스터는 진짜 요청을 보낸다.** dryRun 같은 안전장치가 없다.
- 화면 상단에 이 경고를 상시 표시한다. 운영 API에 POST/PUT/DELETE 보낼 때 주의.

## 설계 포인트
- **4xx/5xx는 실패가 아니다**: 서버가 준 응답을 그대로 보여준다(RestClient.exchange 사용).
  에러 응답 바디에 원인이 담겨 있는 경우가 많아, 이게 테스터의 본분이다.
- **연결 실패와 구분**: DNS 실패·타임아웃 등 전송 자체가 안 된 경우만 `error` 필드로 별도 표시.
- **보호장치**: 타임아웃 기본 15초·상한 30초, 응답 바디 1MB 캡(초과 시 잘림 표시).

## API
```
POST /api/http-test
Body: { url, method, headers{}, bodyType: NONE|JSON|FORM|RAW, body?, contentType?, timeoutSec? }
Resp: { status, statusText, durationMs, headers{}, body, bodyTruncated, error }
```

## 검증
- 검증 로직(URL 스킴, 메서드 화이트리스트, 타임아웃 클램핑 1~30, 1MB 캡) 순수 Java 10종 통과.
- 프론트 빌드 통과 + 목업 백엔드로 3케이스 렌더 확인: 200 성공, 404(응답 그대로), 연결실패(error 구분).
- 실제 외부 호출은 사내망에서 `./mvnw spring-boot:run` 후 확인 필요.

## ✅ 인터페이스로 가져오기 (구현됨)
2xx 응답을 받으면 [인터페이스로 가져오기] 버튼이 나타난다. 테스트한 요청이 그대로 인터페이스 정의가 된다.

**핵심 문제**: 테스터의 바디는 "고정값"이지만(`billNo=B20260712001`), 인터페이스의 바디는
"소스에서 매번 채워질 필드"다. 그래서 바디의 각 키를 어디로 보낼지 사용자가 정한다:

| 역할 | 의미 | 결과 |
|---|---|---|
| **매핑** | 소스 데이터에서 매번 채워질 값 | 필드매핑 탭에 타겟필드로 생성 (소스필드는 사용자가 지정) |
| **고정값** | 매번 같은 값 | `constParams`로 |
| **인증** | 시크릿 | `auth.fields`에 **환경변수 참조**(`${CLIENT_SECRET}`)로 — 실제 값은 DB에 저장 안 함 |
| **제외** | 버림 | — |

**자동 추론**: clientId/clientSecret/token/apiKey 등 시크릿 패턴은 "인증"으로, 나머지는 "매핑"으로
초기 선택된다. 사용자가 조정만 하면 된다.

**안전장치**:
- 가져온 직후 **dryRun=true**로 설정 → 검증 전에 실수로 실제 전송되지 않는다.
- 인증 헤더(Authorization 등)도 환경변수 참조로 변환.
- 편집기로 넘길 때 URL 쿼리가 아니라 router state를 사용 → 시크릿이 주소창/히스토리에 남지 않는다.

**변환 검증**: 키 파싱(JSON/FORM), 역할 자동추론, camelCase→SNAKE_UPPER 환경변수 변환 등 11종 통과.
E2E 렌더 확인: 테스트 → 역할 지정 모달 → 편집기(타겟 설정·매핑 타겟필드 자동 채움)까지.

## 향후 (미구현)
- 요청 히스토리/저장(컬렉션)
- RAW/XML 바디의 필드 분해 (현재는 JSON/FORM만 키 단위 분해 가능)
