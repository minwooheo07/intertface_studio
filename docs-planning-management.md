# 인터페이스 관리도구 기획서

삼천리에너지 SI · IF Engine 관리 UI (코드명: IF Studio)

> 목적: 타사 시스템(더존 ERP·다우오피스·빌링·모뎀 검침 등) 연동 인터페이스를
> SQL 직접 등록 없이 **개발 → 테스트 → 운영 → 문서화**까지 한 화면에서 다룬다.

---

## 1. 배경과 목표

현재 IF Engine은 인터페이스 "실행"은 하지만, 정의를 `IF_MASTER`/`IF_MAPPING`에
SQL로 직접 넣어야 한다. 실행 앞뒤 단계(등록·테스트·문서화)가 비어 있어 현장 생산성이 떨어진다.

- **사용자**: SI 개발자 (JSON/쿼리를 직접 다뤄도 무방). 따라서 화면을 과도하게 추상화하지 않고,
  원문(쿼리·JSON)을 그대로 편집하되 반복작업만 자동화하는 "개발자용 효율 도구"로 만든다.
- **목표 우선순위**: ① 전 영역 균형 ② 신규 인터페이스 개발 속도 ③ 운영 장애 대응·재처리 ④ 문서화·산출물

## 2. 핵심 컨셉 — 인터페이스 생명주기

기획의 뼈대는 하나의 흐름이다. 모든 기능을 이 5단계 위에 배치한다.

```
 ①개발(정의)  →  ②테스트(검증)  →  ③배포(활성화)  →  ④운영(관제/재처리)  →  ⑤문서화(산출물)
     │               │                  │                    │                      │
  복제·템플릿      시험실행          스케줄 등록           장애 큐·재처리        정의서 자동생성
  매핑 자동초안    변환 미리보기      dryRun 해제          알림 규칙             변경이력
```

기존 자산과의 관계: ③④는 이미 상당 부분 구현됨(스케줄러, 대시보드, 단건 재처리).
이번 기획의 신규 비중은 **①개발 ②테스트 ⑤문서화**에 크다.

## 3. 기능 영역별 상세

### A. 인터페이스 개발 (정의/편집) — 개발 속도의 핵심

SQL 없이 IF_MASTER/IF_MAPPING을 CRUD 한다. 개발자가 원문을 다루되, 반복을 걷어낸다.

- **인터페이스 목록**: 검색/필터(소스·타겟 시스템, 채널, 사용여부, 최근 상태). 상태 뱃지.
- **정의 편집기 (탭 구성)**: 기본정보 / 소스 / 타겟 / 필드매핑 / 테스트.
  - 소스·타겟 탭은 채널유형(DB·REST·FILE·SOCKET)에 따라 폼이 바뀐다.
    DB면 datasource·query·markQuery, REST면 url·bodyType·auth·constParams.
    JSON 원문 편집 토글도 제공(개발자 대상이므로).
- **복제(Clone)**: 유사 인터페이스를 복제 후 수정. SI 인터페이스는 패턴 반복이 많아 효과가 크다.
- **템플릿**: 채널 조합별 표준 뼈대. 예) `DB→DB 더존 I/F`, `DB→REST 다우 기안`, `SOCKET→DB 검침수신`.
  신규 생성 시 템플릿을 고르면 SRC_CONFIG/TGT_CONFIG 골격과 권장 매핑이 미리 채워진다.
- **매핑 자동 초안 (킬러 기능)**: 소스 query를 `LIMIT`으로 안전 실행 → 컬럼 목록 추출 →
  IF_MAPPING 행 자동 생성. 소스·타겟 필드명이 같으면 자동 연결, 다르면 개발자가 타겟명만 수정.
  타겟이 DB면 타겟 테이블 컬럼도 인트로스펙션해 후보를 제시.
- **인라인 검증**: 저장 전 JSON 유효성, cron 표현식 유효성, 매핑 타겟필드 중복, 중복방지키 존재 여부 체크.

### B. 테스트 / 검증 — 배포 전 안전장치

- **단건 시험실행 (전 채널 dryRun)**: 소스에서 1건만 뽑아 변환까지 수행하고 **실제 전송은 안 함**.
  결과(원본 → 변환결과)를 나란히 표시. 현재 REST 어댑터에만 있는 dryRun을 어댑터 공통 기능으로 승격.
- **변환 미리보기**: 매핑 편집 중, 샘플 row에 변환룰(CONST/DEFAULT/DATEFMT/CODEMAP)을 실시간 적용해 결과 표시.
- **연결 테스트**: datasource 접속 확인(ping), REST 엔드포인트 도달성 확인. dryRun 로그로 최종 페이로드 검수.

### C. 운영 / 관제 — 장애 대응 (일부 구현됨, 확장)

- **대시보드**(구현됨): 금일 성공/실패/스킵, 인터페이스별 최근 상태.
- **장애 큐 (신규)**: FAIL 건만 모아 인터페이스별·기간별 필터, 동일 오류 메시지 그룹핑으로 원인 파악.
- **일괄 재처리 (신규)**: 조건(인터페이스·기간·오류유형)으로 다건 선택 후 재처리.
  현재는 단건만 가능. 중복방지키로 기성공 건은 자동 skip(엔진에 이미 존재)이라 중복전송 위험 없음.
- **알림 규칙 (신규)**: 인터페이스별 실패 임계치/미수신(지연) 감지 → 메일 또는 다우 알림.
  (엔진의 NotifyService 실연동 = 기존 백로그 T5와 연결)

### D. 문서화 / 산출물 — SI 제출물 자동화

- **인터페이스 정의서 자동생성 (킬러 기능)**: IF_MASTER + IF_MAPPING → **Excel 정의서**.
  포함: IF ID·명·송수신 시스템·채널·주기·중복키·필드매핑표·변환규칙·설명.
  SI 산출물로 그대로 제출 가능. 표준 서식에 맞춰 템플릿화.
- **인터페이스 총괄표 export**: 전체 목록을 Excel/CSV로.
- **변경 이력(버전 스냅샷)**: 정의 수정 시 이전 버전을 스냅샷 보관. 언제·누가·무엇을 바꿨는지 추적.
  운영 중 "이 인터페이스 언제부터 이렇게 됐지?" 추적과 감사 대응.

## 4. 화면 구성 (메뉴 트리)

```
IF Studio
├─ 대시보드            현황 요약 (구현됨)
├─ 인터페이스          목록 → 정의 편집기(기본/소스/타겟/매핑/테스트 탭)
│   └─ [복제] [템플릿에서 생성] [시험실행] [정의서 출력]
├─ 실행 이력           IF_LOG 조회 (구현됨, 필터 강화)
├─ 장애 큐             FAIL 모아보기 + 일괄 재처리 (신규)
├─ 산출물              정의서 생성 / 총괄표 export (신규)
└─ 설정
    ├─ 데이터소스       연동 시스템 접속정보 (application.yml 대체/보완)
    ├─ 알림 규칙        실패·지연 알림 조건 (신규)
    ├─ 코드매핑         CODE_MAP 관리 (백로그 T4)
    └─ 템플릿          인터페이스 템플릿 관리 (신규)
```

## 5. 데이터 모델 확장

기존 3테이블(IF_MASTER/IF_MAPPING/IF_LOG)에 더해 신규 4종.

| 테이블            | 용도                                                        |
|-------------------|-------------------------------------------------------------|
| IF_TEMPLATE       | 인터페이스 템플릿 (채널조합별 SRC/TGT_CONFIG 골격 + 기본매핑) |
| IF_CHANGE_LOG     | 정의 변경 이력 (IF_ID, 버전, 변경자, 변경시각, 스냅샷 JSON)   |
| IF_ALERT_RULE     | 알림 규칙 (IF_ID, 조건유형, 임계치, 통보채널, 수신자)         |
| CODE_MAP          | 코드매핑 (GROUP_ID, SRC_CODE, TGT_CODE) — T4와 공용          |

선택: `IF_EXEC_SUMMARY`(실행 배치별 요약)를 두면 통계·대시보드 조회가 빨라진다.
초기엔 IF_LOG 집계로 충분, 데이터 늘면 도입.

## 6. API 설계 개요

| 메서드 | 경로                                   | 설명                        |
|--------|----------------------------------------|-----------------------------|
| GET    | `/api/interfaces`                      | 목록 (구현됨, 필터 확장)     |
| GET    | `/api/interfaces/{id}`                 | 상세 (마스터+매핑)           |
| POST   | `/api/interfaces`                      | 신규 등록                    |
| PUT    | `/api/interfaces/{id}`                 | 수정 (+ 변경이력 스냅샷)      |
| POST   | `/api/interfaces/{id}/clone`           | 복제                         |
| POST   | `/api/interfaces/{id}/mappings:draft`  | 매핑 자동초안 (쿼리 인트로스펙션) |
| POST   | `/api/interfaces/{id}/test`            | 단건 시험실행 (dryRun)       |
| POST   | `/api/interfaces/{id}/run`             | 즉시 실행 (구현됨)           |
| GET    | `/api/interfaces/{id}/doc.xlsx`        | 정의서 Excel 생성            |
| GET    | `/api/logs`                            | 로그 조회 (구현됨)           |
| POST   | `/api/logs:reprocess`                  | 일괄 재처리 (조건 기반)       |
| CRUD   | `/api/templates`, `/api/alert-rules`, `/api/code-maps` | 설정 관리 |

## 7. 권한 / 감사

개발자 대상 내부도구라 초기엔 단순하게. 단, 운영 반영이 되는 동작은 이력을 남긴다.

- 정의 변경·삭제·일괄 재처리·dryRun 해제(실전송 전환)는 IF_CHANGE_LOG 또는 별도 감사로그 기록.
- 시크릿(다우 clientSecret 등)은 화면·API·로그에 평문 노출 금지. 환경변수 주입 원칙 유지(엔진에 마스킹 구현됨).
- 권한 등급(조회/편집/운영)은 2단계 이후 도입 여지만 남겨둔다.

## 8. 개발 로드맵

기존 백로그 T2를 이 기획으로 구체화. 각 단계는 독립적으로 배포 가능하게 끊는다.

| 단계 | 범위                                                          | 산출 효과            |
|------|---------------------------------------------------------------|----------------------|
| P1   | 인터페이스 CRUD + 매핑 편집기 (SQL 없이 등록/수정)             | SQL 등록 탈피         |
| P2   | 개발 가속: 복제 · 템플릿 · 매핑 자동초안 · 단건 시험실행       | 신규 개발 속도 ↑ (목표②)|
| P3   | 운영: 장애 큐 · 일괄 재처리 · 알림 규칙                        | 장애 대응 ↑ (목표③)   |
| P4   | 문서화: 정의서 Excel 자동생성 · 총괄표 · 변경이력              | 산출물 자동화 (목표④) |
| P5   | 부가: 코드매핑 UI(T4) · 통계/추이 · 권한                       | 완성도               |

P1→P2가 개발 생산성의 본체, P3가 운영, P4가 산출물. 균형 목표(①)에 맞춰 P1~P4를 모두 범위에 포함.

## 9. 비범위 (이번에 안 하는 것)

과설계를 막기 위해 선을 긋는다.

- **드래그앤드롭 비주얼 매핑 편집기**: 개발자 대상이므로 표(그리드) 기반 매핑으로 충분. 노코드 GUI는 과함.
- **BPM/오케스트레이션**(여러 인터페이스를 조건분기로 엮는 워크플로우): 별도 주제. 지금은 단위 인터페이스에 집중.
- **실시간 스트리밍/메시지큐 연동**: 현재 배치·주기 실행 모델 유지. 필요 시 별도 검토.
- **멀티테넌시/외부 공개**: 사내 SI 내부 도구 범위.
